Frontend Frameworks

(HTML/CSS)

What are they?

A collection of CSS (less/sass), JS, and markup that follow patterns and create a convention for building projects.

Why use them?

  • Structure for frontend organization
  • Extendable baseline for commonly used patterns
  • Create easily reusable elements from one project to another
  • Rapid prototyping/iteration
  • Bringing a project up to speed quickly, so more time can be spent on the things that make it different and unique
  • Thousands of hours of testing have already been done for you!
  • A team of hundreds of developers is backing your code
  • Community

What ones exist?

  • HTML5 Boilerplate
  • Zurb Foundation
  • Twitter Bootstrap
  • Grid only frameworks: Singularity, Susy, Neat

So what's the difference?

Their intention and function is really in each of their names.

HTML5 Boilerplate

  • Worth noting - widely used
  • Function: Super slim starting point for an HTML project. Some normalization, and avoids the bugs.

Bootstrap

  • Bootstraps your web app
  • Has more opinions about markup, style, and function
  • Tends toward more handsoff front end devs
  • Takes care of doing a lot for you
  • May often need to "unstyle" conventions put in place by bootstrap
  • Written in Less, ported to Sass

Foundation

  • A set of tools to build a web app upon
  • Doesn't look as pretty or styled out of the box, but the intention is that you are not going to be using it out of the box
  • Very extendable baselines for common patterns
  • Written in Sass originally
  • More comprehensive grid system
  • Very extensive settings system and modular sass setup

Common concerns and resons maybe not to use such an involved framework

  • Classes
  • Size/bloat
  • Limiting structure

Ewww, the classes...I like my internet semantic!

  • Foundation comes with a pretty comprehensive set of mixins for each component
  • In fact, you can even turn off all of the default css class output for each component and use it as a mixin library if you like
Example using grid mixins 

.some-informative-class-name {
  @include grid-row();
  .another-very-semantic-class {
    @include grid-col(8);
  }
  .content-right {
    @include grid-col(4);
    @include i-only-use-mixins(true);
  }
}

What about the bloat?

  • Broken up into modules that can be included or excluded
  • All components not minified ~200k
  • Just essentials (typography, grid, buttons, a couple others) ~60k
  • Adding F5 to e3_zen without any other additional style compiles to a style.css file at 208kb
  • If you change the "Add html classes" setting to false across the board, it compiles to 16kb, same as base compile size
  • Use it as a mixin library
  • All in all, less than most images, cached after first page visit
  • For our purposes with Drupal, I'd argue in most cases, size is a non-issue

The site design doesn't fit into a grid! (or the established setup of some other css component)

  • Forcing things into the structure Foundation provides is where things can start getting frustrating
  • If it doesn't fit, don't use it, or modify it, or extend it
  • The components are designed to be the base bits necessary to accomplish certain tasks, generally, modifying the basic variables via the _settings.scss file, and extending upon that

When does Foundation fall short?

  • Large team maintaining an internal service/product
  • If the project really needs to completely different, fully functioning row based grid layouts
  • Foundation doesn't support this out of the box
  • Can probably provision it without too much difficulty using the mixins in the _grid.scss file

I don't like it

  • A perfectly good reason not to use it

The Details (F5.x)

  • Foundation uses bower to manage version updates
  • Uses REM (root em) units
  • No IE8 support
  • A well featured _settings.scss file
  • Built in SASS, easily componentized
  • SASS components and JS utilities, some work together
  • If IE8 support is needed (maybe NEVER again???) you can use Foundation 3 for a grid system that supports IE8 (very old, but still a very well featured grid system, I've used it before)

Noteworthy Components

  • Typography

  • The grids - row and block
  • Top-bar
  • Off-canvas
  • Modals

Off-Canvas

  • Has a great starting point for the off canvas pattern
  • By default, docs
  • Can be extended relatively easily build upon functionality
  • Menu overlap instead of push is an option
  • The pattern creates the most accessible way to recreate native app behavior with window, and either nav, all scrolling separately
  • Given hooks to trigger menu, callbacks, etc
  • NJH, GPB

Foundation with Drupal

  • Foundatoin relies on sass, but not necessarily ruby, can use ruby gem, or libsass with a js compiler, gulp or grunt
  • Pretty much everything can be included with mixins and vars, it's pretty easy to target whatever you need even with the drupal markup jungle
  • Multiple drupal menus can easily collapse into one mobile menu
  • Top bar has built in features for touch device mobile menus, ie - repeat parent menu item

Setup

  • For ruby within our current theme setup
  • Add a bower_components directory to theme
  • Inside add a folder "foundation" containing bower_foundation components
  • From thecompass template, grab the bower files, adjust gemfile accordingly, and add the line to import the bower path into the config.rb file
  • Add the SCSS to the appropriate folder, and import "app" and "settings"
  • Add the foundation.min.js file if desired

Menus

  • Multiple drupal menus can easily collapse into one mobile menu
  • Top bar has built optionals in features for touch device mobile menus, ie - repeat parent menu item, back
  • Use template_menu_link() to add the default Foundation classes if desired

e3_zen_menu_link()



function e3_zen_menu_link(&$variables) {
  $element = $variables['element'];
  $sub_menu = '';

  // if has children add the appropriate foundation menu class
  if ($element['#original_link']['has_children']) {
    $element['#attributes']['class'] = 'has-dropdown';
  }

//  $element['#attributes']['data-menu-parent'] = $element['#original_link']['menu_name'] . '-' . $element['#original_link']['depth'];

  // render sub-menus with the foundation dropdown class
  if ($element['#below']) {
    $sub_menu = '
    ' . drupal_render($element['#below']) . '
';
  }

  $output = l($element['#title'], $element['#href'], $element['#localized_options']);

  return '' . $output . $sub_menu . "";
}
